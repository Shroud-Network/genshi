# Shroud-Honk: Improvements

## Tier 1: Improvements that fall out of the architecture naturally (low effort once baseline ships)

* **Custom gates for Poseidon2.** UltraHonk supports custom gates beyond standard addition/multiplication. Barretenberg already has a hand-coded Poseidon2 gate that evaluates the full round function in a single gate instead of decomposing it into individual field operations. If you implement a similar custom gate in shroud-honk, your Poseidon2 constraints drop further -- potentially 30-50% cheaper per hash call than even the baseline estimate of ~150 constraints. This compounds across Merkle paths (10 hashes), note commitments (1 hash each), and nullifier derivation. It's a circuit-level optimization, not a protocol change.
* **Lazy SRS streaming.** The baseline plan loads the full SRS into WASM memory before proving. Barretenberg recently shipped configurable SRS sizing for browser environments. You could implement progressive SRS loading -- download only the SRS points needed for the specific circuit, and stream them during polynomial commitment evaluation rather than front-loading the entire download. This cuts the initial page load latency for first-time users from "download N MB then prove" to "start proving while downloading."
* **Proof compression via recursive wrapping.** Your ~2KB UltraHonk proof is fine for Avalanche and Solana where calldata/transaction data is cheap. But if you ever target Ethereum L1, that's 2KB of calldata at ~16 gas per byte = ~32K gas just for proof data, on top of ~300-500K verification gas. A recursive UltraHonk-to-Groth16 wrapper compresses the final on-chain proof to 128 bytes and drops verification to ~200K gas. Aztec does exactly this for their L1 settlement. The inner proof (UltraHonk) stays fast to generate client-side; the outer proof (Groth16) is generated server-side by a sequencer/relayer. This is additive -- it doesn't change the client-side architecture at all.

## Tier 2: Throughput improvements (medium effort, major protocol impact)

* **Batch insertion with proof aggregation.** This is the real answer to the sequential Merkle tree bottleneck (issue 1.5 in the blueprint). Instead of one proof per transfer updating the tree once, a relayer collects N pending transfers, generates one aggregated proof that validates all N transfers simultaneously, and submits one on-chain transaction that updates the tree N times. Per-user on-chain cost drops by 1/N. The circuit needs to be designed to support variable batch sizes (or a fixed batch size with dummy operations for padding). This is the single highest-impact throughput improvement available. Aztec's rollup does this. For Shroud's B2B API model, this is especially natural -- API clients submit transfers to the relayer, the relayer batches and proves.
* **Recursive proof aggregation via IVC (Nova/HyperNova).** This is the more ambitious version of batching. Each user generates their own proof locally (fast, parallel, no coordination needed). A sequencer takes N individual proofs and folds them into a single recursive proof using an IVC scheme. Microsoft's Nova library supports the BN254/Grumpkin curve cycle natively GitHub, which is exactly the cycle shroud-honk is built on. The PSE fork of Nova also supports BN254/Grumpkin GitHub. The Sonobe library provides a modular framework for folding with multiple schemes including Nova and HyperNova, with tools for generating a final SNARK proof verifiable on Ethereum's EVM GitHub. This eliminates the sequential bottleneck entirely because proof generation is fully parallelized across users and only the final aggregated proof touches the chain. The engineering cost is significant (2-3 months on top of the baseline), but the throughput gain is multiplicative.
* **Optimistic note insertion.** A lighter-weight concurrency solution: let users generate proofs against a pending root (not yet confirmed on-chain) with a fallback mechanism to re-prove if the pending root gets invalidated by a conflicting insertion. This handles moderate concurrency without the complexity of recursive aggregation. The SDK tracks pending insertions locally and detects conflicts. Failed proofs are automatically regenerated against the updated root. This is implementable in the SDK layer with minimal circuit changes.
